## Decisions made this session

1. **Feature allowlist moved from a file to Postgres.** `config/allowlist.txt` is baked into the Docker image at build time (`COPY config ./config` in `Dockerfile`) ŌĆö an admin UI editing that file would never persist on Cloud Run's immutable containers. Moved to a new `allowlist` table (same Neon DB as `documents`), with a 30-second in-memory cache in `lib/feature-gate.ts` and a file fallback only when Postgres is unreachable (local dev).
2. **Wildcard access removed ŌĆö invite-only.** The old `* : 2` line in `config/allowlist.txt` gave every Google account One-Pager access. Per explicit user decision, this is gone; every user must be explicitly listed. Guests (e.g. shmuel@, shahar@) are added by an admin.
3. **New admin UI at `/admin/allowlist`** (page + `AllowlistClient.tsx`) and API (`GET/POST /api/admin/allowlist`, `DELETE /api/admin/allowlist/[email]`), gated by the existing `isAdmin()` check ŌĆö same pattern as the version-rollback feature.
4. **Cloud SQL was dropped from the plan.** Originally scoped as the Neon replacement; the user decided partway through to drop it in favor of a non-relational approach for document storage specifically (see "GCS design ŌĆö not yet implemented" below). This was a mid-session pivot ŌĆö Cloud SQL provisioning was never completed and the `--add-cloudsql-instances` flag added to `deploy.yml` was subsequently removed.
5. **IAM was fixed by self-granting roles**, with the user's explicit authorization each time (never done silently):

   - `roles/logging.viewer` ŌĆö needed to actually read the stack trace behind a production 500 (see below).

## The production bug (and the actual root cause)

After deploying, `/api/documents` and `/api/admin/allowlist` both 500'd with ~5 second latency. Investigation path (all real, in order ŌĆö nothing skipped):

1. Assumed Neon IP allowlist or cold-start (`connect_timeout` default is 5s, matched the observed latency almost exactly) ŌĆö added `connect_timeout=15` to the secret. Did not fix it.
2. Got `logging.viewer` access (self-granted, authorized) and read the actual stack trace: `Error: getaddrinfo EAI_AGAIN base`. The app was trying to resolve a literal hostname `"base"` ŌĆö not a Neon problem at all.
3. Root cause: `.env.local` stores `POSTGRES_URL="postgresql://...";` **with literal double quotes** as part of the file's bash-style syntax. The original `gcloud secrets create POSTGRES_URL --data-file=-` piped the value through `grep '^POSTGRES_URL=' .env.local | sed 's/^POSTGRES_URL=//'`, which stripped the variable name but not the quote characters. `pg`'s connection-string parser choked on the leading `"` and fell through to a garbage parse, landing on hostname `base`.
4. Fixed by creating secret version 3 with a small Node one-liner that explicitly strips a leading/trailing `"` before writing to stdout, piped directly into `gcloud secrets versions add`.
5. **Verified the fix on an isolated tagged revision (0% traffic) before routing production traffic to it** ŌĆö did not repeat the earlier mistake of assuming a change worked.

**Lesson for next time**: never assume `grep VAR=file | sed 's/VAR=//'` gives you a clean value if the source file might quote its values. Always print (or check the length/structure of) a secret's masked shape before trusting it's correct, and always verify a fix on a 0%-traffic tagged revision before cutting over.

## Next session ŌĆö what's already decided (don't re-litigate)

From the design conversation, these are settled and should NOT be re-asked:
- **GCS, not Drive**, for the JSON documents themselves (faster, cheaper at this scale, IAM-based access control). Drive is used only for manual DOCX/HTML exports, on-demand (not automatic on publish).
- **One bucket, three prefixes**: `one-pager/`, `prd/`, `prd-runs/` (the last for ephemeral PRD pipeline run state).
- **Filename pattern**: `{productName} - {docType} - {Draft|Published} {version}.json`, one file per document (not per version), overwritten on save/publish. Version history lives *inside* the JSON (`versionHistory` array), same shape as today's Postgres column.
- **A stable UUID `id` field inside the JSON is the real lookup key** ŌĆö filenames are a human-readable projection. A per-prefix `_index.json` maps id ŌåÆ current filename to avoid full-bucket listings.
- **Concurrency**: single sequential writer per document (confirmed by the user) ŌĆö plain read-modify-write with a GCS generation-match precondition is sufficient, no locking/retry queue needed.
- **PRD pipeline run state also moves to GCS** (not just the final PRD) ŌĆö the user chose the larger-scope option here, meaning `pipeline_runs`/`prd_frames`/`prd_documents` tables all go away, replaced by `prd-runs/{runId}.json`.
- **MRD is explicitly out of scope.** It isn't released yet. Do not wire it to anything as part of this work, and do not assume the old 12-section `app/mrd-generator` code needs to be finished first ŌĆö the user was clear that One-Pager is the only tool that needs to actually work right now.
- **Allowlist stays in Postgres** ŌĆö already migrated, tested, working. Do not move it to GCS.
- One open question that was asked but never answered: whether publishing a One-Pager needs admin sign-off, or whether the existing owner-can-edit-and-publish-their-own-doc model stays as-is. Defaulted to "no new gate" for now since it wasn't confirmed ŌĆö flag this again if it becomes relevant.
